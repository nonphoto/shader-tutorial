Shaders

By Jonas Luebbers

Shaders are for drawing things.

They are for drawing things a very specific kind of way.

This way happens to be difficult, but fast.

They can be intimidating at first, but using them on the web is easier then you might expect.

Why bother?

Most of the time, don't bother.

We already have things like p5.js and three.js which are often

  1. Better
  2. Easier
You might be tempted to use the speed of shaders everywhere, but using a framework usually makes it easier to get a fast running program.

Shaders are for when you need to DIY.

You are responsible for a lot more of the rendering pipeline.

This means more work, but it can have impressive results:

  1. Full control over every pixel.
  2. Dynamic and non-geometric textures.
  3. Hardware-accelerated rasterization.
  4. Ludicrous speed (sometimes).
Rasterization is the process of turning 3d geometry into a 2d image.

How do CPUs work?

CPUs are the main processing unit in most computers.

Many also have a GPU, which we will get to in a second.

They do one big task at a time, in sequence.

Some CPUs have multiple threads that each do their own task.

Part of the computation is also dedicated to deciding what to do next.

Why are GPUs fast?

Rendering an image requires a very large number of small tasks.

Instead of doing one at a time, thousands of threads process a chunk of tasks at once.

These smaller threads can't communicate between each other.

They can't decide which tasks to do.

WebGL is a technology that allows you to run GPU programs on the web.

GLSL is the language that GPU programs are typically written in.

On the web we use Javascript running on the CPU to tell WebGL to run some programs that we wrote in GLSL on the GPU.
